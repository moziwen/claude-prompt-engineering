# 04. 安全, 压缩, 输出 为什么也属于 Prompt Engineering

## 学习目标

阅读本篇后，建议能够理解以下问题：

- 为什么安全约束不应只放在运行时权限系统中。
- Claude Code 如何把 compact 变成可续跑的上下文重建机制。
- 为什么用户可见输出应该被视为 prompt 设计的一部分。
- 这些机制如何共同塑造产品级体验。

## 1. 安全规则不仅存在于权限系统中

Claude Code 具有完善的权限与审批机制，但它并没有把安全完全交给运行时代码。相反，system prompt 和工具 prompt 也承担了显著的安全教育与行为约束职责。

从 `src/constants/prompts.ts` 可以观察到，它会明确提醒模型：

- 工具结果可能包含外部数据。
- 工具结果可能包含 prompt injection。
- 风险动作要谨慎处理。
- 被用户拒绝后不要机械重试。

### 1.1 为什么需要 prompt 层安全

因为很多风险不是一次权限检查就能完全解决的。例如：

- 模型可能被网页、工具输出或第三方内容误导。
- 模型可能在一次拒绝后换一种形式继续尝试相同动作。
- 模型可能把高风险行为当作“最快捷的解决方案”。

因此，Claude Code 的设计是双层的：

- 运行时代码负责硬约束。
- prompt 负责软约束和行为引导。

## 2. 风险动作规则为什么要写得具体

`Executing actions with care` 这一段非常值得学习。它没有停留在抽象的“请注意安全”，而是明确要求模型区分不同动作的风险等级，例如：

- 本地、可逆的小动作通常可以直接做。
- 难以回滚、影响共享系统、对外可见的动作需要谨慎。
- 不要用 destructive action 当作捷径。
- 看到陌生状态时应先调查，而不是先删除。

### 2.1 这种写法的价值

它把“谨慎”从价值观层面降落到了操作层面。对 Agent 系统来说，这类具体规则远比泛化提醒更有效。

## 3. “被拒绝后怎么办”为什么值得单独设计

Claude Code 非常强调一条行为规则：

- 用户拒绝工具调用后，不要原样重试。

这条规则看似简单，但它直接决定了系统是“协作者”还是“固执的自动机”。

Claude Code 要求模型在被拒绝后：

- 先判断用户拒绝的原因。
- 调整策略。
- 真正不理解时再去提问。

这使得系统在权限受限场景中仍能维持协作感，而不会不断碰撞同一面墙。

## 4. compact 不是附属功能，而是续航机制

`src/services/compact/prompt.ts` 展示了 Claude Code 对 compact 的理解：它不是“顺手总结一下”，而是“为后续继续工作重建上下文”。

compact prompt 会要求模型保留：

- 用户意图
- 关键技术概念
- 涉及的文件和代码段
- 错误与修复
- 当前工作进度
- 下一步行动

### 4.1 为什么这不是普通摘要

普通摘要面向阅读。
Claude Code 的 compact 面向继续执行。

两者的目标完全不同。前者关注概括质量，后者关注工作连续性。

### 4.2 `<analysis>` 与 `<summary>` 的设计意义

Claude Code 要求模型先写 `<analysis>`，再写 `<summary>`。这个设计的价值在于：

- 让模型在输出正式摘要前先整理信息。
- 把中间草稿从最终上下文中剥离出去。
- 提高压缩结果的结构化程度与完整性。

## 5. 输出体验为什么进入 prompt 层

对终端类产品而言，用户往往看不到完整的 tool use、内部状态和中间调度。用户实际接触到的，是模型最终输出的文字。

因此 Claude Code 在 prompt 中专门约束：

- 何时给进展更新
- 何时直接给结论
- 如何避免重复用户原话
- 如何引用文件与代码位置
- 如何保持文本简洁而不丢失信息

这说明它把用户可见输出视为系统行为的一部分，而不是 UI 层的附属美化。

## 6. output style 说明“风格”本质上也是行为控制

`src/constants/outputStyles.ts` 中的 Explanatory 和 Learning 两种 output style 很能说明问题。

它们并不是简单改变语气，而是通过 prompt 直接改变模型行为，例如：

- 是否需要输出教学性 insight
- 是否需要邀请用户写一小段代码参与实现
- 是否保留完整 coding instructions

这意味着 Claude Code 将“风格”视为 prompt architecture 的组成部分，而不是事后修辞。

## 7. 三个维度如何共同塑造产品质量

从系统设计角度看，这三件事分别解决了不同问题：

- 安全：防止系统做错事或越界。
- compact：防止系统在长会话中漂移。
- 输出：防止系统虽然做对了事，但让用户感到难用。

如果缺少其中任意一项，系统都会出现明显缺陷：

- 没有安全，系统容易越权。
- 没有 compact，系统会在长会话中逐渐失真。
- 没有输出规范，系统即使功能正确，也可能缺乏可用性。

## 8. 复用建议

如果要在自己的 Agent 产品中借鉴 Claude Code 的经验，建议至少做到以下三点：

### 8.1 给高风险动作单独写规则段

不要只写“注意安全”，而要写：

- 哪些动作可直接做
- 哪些动作需要确认
- 被拒绝后如何调整
- 哪些动作不能作为省事捷径

### 8.2 给 compact 单独设计 prompt

不要把 compact 交给一个通用摘要器。应该明确规定它要保留哪些工作状态。

### 8.3 给用户可见输出单独设计规范

尤其是终端、IDE、协作式 Agent 产品，输出质量直接决定产品观感。

## 9. 本篇结论

Claude Code 的 prompt engineering 并不只面向“让模型更聪明”，还同时面向三件事：

- 风险控制
- 长会话稳定性
- 协作体验

这正是产品级 prompt 设计与 demo 级 prompt 设计的重要区别。
